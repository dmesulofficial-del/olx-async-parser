Висновок щодо реалізації парсингу OLX
Для виконання завдання було обрано асинхронний підхід на базі бібліотеки Playwright, що дозволило ефективно поєднати швидкість роботи та можливість взаємодії з динамічним контентом сайту.

1. Метод збору даних
Парсинг було розділено на два етапи для оптимізації ресурсів:

Первинний збір: За допомогою методів Playwright було проскановано головну сторінку каталогу. Ми витягли посилання на кожне оголошення, ціну та локацію безпосередньо з карток списку (div[data-testid="l-card"]).

Глибокий аналіз: Для отримання специфічних даних (поверх, площа) скрипт асинхронно відкриває кожне оголошення. Оскільки OLX використовує динамічну структуру тегів, було застосовано гнучкий пошук у контейнері ad-parameters-container.

2. Технічні рішення та виклики
Асинхронність та стабільність: Щоб уникнути блокування з боку анти-бот системи OLX, використано asyncio.Semaphore(1). Це дозволило обробляти дані в один потік, але з мінімальними затримками, що імітують поведінку реального користувача.

Обробка динамічної верстки: Під час розробки було виявлено, що характеристики об’єктів завантажуються через JavaScript після рендерінгу сторінки. Ми вирішили це за допомогою wait_until="domcontentloaded" та невеликих пауз (wait_for_timeout), що гарантує наявність даних у DOM перед їх зчитуванням.

Структурування: Дані збираються у словник (Map) за ключовими словами ("поверх", "площа" тощо), що робить скрипт стійким до зміни порядку характеристик у списку на сайті.

3. Експорт та безпека
Google Sheets API: Дані автоматично очищуються та записуються в Google Таблицю через бібліотеку gspread. Реалізовано функцію масового запису (append_rows), що значно швидше за запис кожного рядка окремо.

Безпека даних: Усі чутливі дані (JSON-ключі сервісного акаунта) винесені в окремий файл creds.json, який додано до .gitignore, щоб запобігти їх витоку у відкритий репозиторій.
